session、cookie、token

发表于 2021-09-05 更新于 2021-09-11 分类于 HTTP 阅读次数：

一.cookie和session

1.客户端发送请求

2.服务器接收到请求建立一个session，并在请求头中包含一个set-cookie，其中包含sessionID

3.客户端将数据存储在cookie，sessionID包含在cookie中

4.服务器根据sessionID找到session并检查数据(第一次建立数据)，返回response

XSRF就是利用用户在未关闭A网站的前提下去访问攻击者B网站，B网站发送request请求用户访问A网站并拿到用户在A网站的数据

Set-Cookie属性：

  • ❤️持久性 Cookie 的生命周期取决于过期时间（Expires）或有效期（Max-Age）指定的一段时间。
  • 🧡有两种方法可以确保 Cookie 被安全发送，并且不会被意外的参与者或脚本访问：Secure(只能在https中携带) 属性和HttpOnly(cookie中不允许有js代码，预防xss攻击) 属性。

  • 💛Domain 和 Path 标识定义了Cookie的作用域：即允许 Cookie 应该发送给哪些URL。

    Domain 指定了哪些主机可以接受 Cookie。如果不指定，默认为 origin，不包含子域名。

    Path 标识指定了主机下的哪些路径可以接受 Cookie，若为/api则/api/dos,/api都能匹配到

  • 💚SameSite规定跨站请求时不会发送，减少CSRF攻击

Cookie前缀：

cookie 机制的使得服务器无法确认 cookie 是在安全来源上设置的，甚至无法确定 cookie 最初是在哪里设置的。

如果 cookie 名称具有此前缀，则仅当它也用 Secure 属性标记，是从安全来源发送的，不包括 Domain 属性，并将 Path 属性设置为 / 时，它才在 Set-Cookie 标头中接受。这样，这些 cookie 可以被视为 “domain-locked”。

如果 cookie 名称具有此前缀，则仅当它也用 Secure 属性标记，是从安全来源发送的，它才在 Set-Cookie 标头中接受。该前缀限制要弱于 __Host- 前缀。

获取或设置cookie

设置：document.cookie = “‘name = yjh’”

获取: const cookie = document.cookie，只能获取键值对

Navigator.cookieEnabled

用来检查浏览器是否开启了cookie，返回布尔值

二.token

组成

  • uid: 用户唯一身份标识
  • time: 当前时间的时间戳
  • sign: 签名, 使用 hash/encrypt 压缩成定长的十六进制字符串，以防止第三方恶意拼接
  • 固定参数(可选): 将一些常用的固定参数加入到 token 中是为了避免重复查库

token是无状态的，服务器不会保存用户相关的数据，而是每次用户请求服务器去检查,token不会直接加在请求头中，这样就可以避免CSRF攻击了

存放

token在客户端一般存放于localStorage，cookie，或sessionStorage中。在服务器一般存于数据库中

token认证流程

token 的认证流程与cookie很相似

  • 用户登录，成功后服务器返回Token给客户端。
  • 客户端收到数据后保存在客户端
  • 客户端再次访问服务器，将token放入headers中
  • 服务器端采用filter过滤器校验。校验成功则返回请求数据，校验失败则返回错误码

三.总结

  • session存储于服务器，可以理解为一个状态列表，拥有一个唯一识别符号sessionId，通常存放于cookie中。服务器收到cookie后解析出sessionId，再去session列表中查找，才能找到相应session。依赖cookie
  • cookie类似一个令牌，装有sessionId，存储在客户端，浏览器通常会自动添加。
  • token也类似一个令牌，无状态，用户信息都被加密到token中，服务器收到token后解密就可知道是哪个用户。需要开发者手动添加。
  • jwt只是一个跨域认证的方案

JWT: 客户端登陆传递信息给服务端，服务端收到后把用户信息加密（token）传给客户端，客户端将token存放于localStroage等容器中。客户端每次访问都传递token，服务端解密token，就知道这个用户是谁了。通过cpu加解密，服务端就不需要存储session占用存储空间，就很好的解决负载均衡多服务器的问题了
