4.session、cookie、token

发表于 2021-09-05 分类于 HTTP

一.cookie和session

1.客户端发送请求

2.服务器接收到请求建立一个session，并在请求头中包含一个set-cookie，轻重包含sessionID

3.客户端将数据存储在cookie，sessionID包含在cookie中

4.服务器根据sessionID找到session并检查数据(第一次建立数据)，返回response

XSRF就是利用用户在未关闭A网站的前提下去访问攻击者B网站，B网站发送request请求用户访问A网站并拿到用户在A网站的数据

二.token

组成

  • uid: 用户唯一身份标识
  • time: 当前时间的时间戳
  • sign: 签名, 使用 hash/encrypt 压缩成定长的十六进制字符串，以防止第三方恶意拼接
  • 固定参数(可选): 将一些常用的固定参数加入到 token 中是为了避免重复查库

token是无状态的，服务器不会保存用户相关的数据，而是每次用户请求服务器去检查,token不会直接加在请求头中，这样就可以避免CSRF攻击了

存放

token在客户端一般存放于localStorage，cookie，或sessionStorage中。在服务器一般存于数据库中

token认证流程

token 的认证流程与cookie很相似

  • 用户登录，成功后服务器返回Token给客户端。
  • 客户端收到数据后保存在客户端
  • 客户端再次访问服务器，将token放入headers中
  • 服务器端采用filter过滤器校验。校验成功则返回请求数据，校验失败则返回错误码

三.总结

  • session存储于服务器，可以理解为一个状态列表，拥有一个唯一识别符号sessionId，通常存放于cookie中。服务器收到cookie后解析出sessionId，再去session列表中查找，才能找到相应session。依赖cookie
  • cookie类似一个令牌，装有sessionId，存储在客户端，浏览器通常会自动添加。
  • token也类似一个令牌，无状态，用户信息都被加密到token中，服务器收到token后解密就可知道是哪个用户。需要开发者手动添加。
  • jwt只是一个跨域认证的方案

JWT: 客户端登陆传递信息给服务端，服务端收到后把用户信息加密（token）传给客户端，客户端将token存放于localStroage等容器中。客户端每次访问都传递token，服务端解密token，就知道这个用户是谁了。通过cpu加解密，服务端就不需要存储session占用存储空间，就很好的解决负载均衡多服务器的问题了

3.http状态码

常见的Http状态码及其含义

1XX

100 Continue 继续，一般在post请求时，在http header发送完之后服务器会返回此状态码，之后发送具体参数信息。

101 Switching Protocols 升级协议

2XX

200 OK 请求成功，服务器返回相关资源

201 Create 请求成功，服务器创建了新的资源

202 Accepted 服务器接受了请求，但还未处理完成

3XX

301 permanently 资源已被永久移动到了新位置，返回信息包括新的URI，浏览器会自动跳转到新的URI

302 Found 临时重定向,资源被临时移动了，应该继续使用原来的URI

303 See Other 查看其它地址。与301类似。使用GET和POST请求查看

304 Not Modified 此次请求资源与上一次请求未发生变化

4XX

400 Bad Request 客户端请求的格式异常，服务器无法理解

401 Unauthorized 用户未授权

402 Payment Required 留作备用

403 Forbidden 用户禁止访问

404 Not Found 请求资源不存在

5XX

500 Internal Server Error 最常见服务器内部错误

501 Service Unavailable 由于超载或系统维护，服务器无法处理客户端请求
